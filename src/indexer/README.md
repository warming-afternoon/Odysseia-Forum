# 🏗️ Indexer 模块

## 📖 简介
`indexer` 模块的作用是扫描指定 Discord 论坛频道中的所有帖子，解析其内容、附件、标签及作者信息，并将其持久化到数据库中。

---

## ⚙️ 核心工作原理

### 1. 生产者-消费者模式
索引过程被拆分为两个异步阶段，以实现并行处理：
- 生产者 (Producer): 负责遍历论坛频道。
    - 首先扫描缓存中的活跃帖子 (`channel.threads`)。
    - 接着使用分页请求扫描已归档的帖子 (`channel.archived_threads`)。
    - 将发现的所有帖子对象放入一个 `asyncio.Queue` 队列。
- 消费者 (Consumer): 负责实际的数据处理。
    - 从队列中获取帖子，调用 `SyncService` 进行标准化解析。
    - 使用 `asyncio.Semaphore` 控制并发数，防止触发 Discord API 的限流。

### 2. 标签预同步机制
由于 SQLite 在高并发写入新标签时可能会发生锁竞争或唯一约束冲突，索引器在启动生产/消费任务前，会先触发 `pre_sync_forum_tags`。它会一次性获取论坛频道定义的所有标签并写入数据库，确保后续消费者处理帖子时，标签记录已经存在。

### 3. UI 实时监控
索引启动后，会发送一个带有实时进度条的 Embed：
- 交互控制: 允许管理员在 UI 上实时暂停或取消任务。
- 错误汇总: 自动收集并展示处理失败的帖子 ID 及其原因，方便后续排查。

---

## 📂 目录结构

- `cog.py`: 模块入口。包含 `/构建索引` 命令、生产者/消费者逻辑、以及 Discord 频道更新事件监听器（用于自动刷新标签缓存）。
- `views.py`: 索引仪表板 UI 类。负责处理 UI 更新循环、按钮交互以及 Embed 渲染。

---

## 💡 设计细节

### 1. 应对 Discord Interaction Token 过期
Discord 的交互令牌有效期只有 15 分钟。如果索引任务非常庞大（例如有数千个帖子），UI 更新任务会：
- 记录 `start_time`。
- 在 14.5 分钟（870秒）后自动停止 `updater_loop`。
- 索引任务完成后，如果发现 Token 已过期，将不再尝试编辑原消息，而是通过 `interaction.followup.send` 发送一条包含最终结果的新消息。

### 2. 任务调度与优先级
- 索引器中的所有 API 调用（如获取归档列表、更新 UI）都通过全局的 `api_scheduler` 进行。
- 索引过程属于“后台任务”，使用较低的优先级（5-10），以确保不影响用户的即时搜索体验。
