# 🧹 Auditor 模块

## 📖 简介
`auditor` 模块负责审计数据库的数据，通过拉取对应 discord 信息进行同步，并负责清理长期失效的“幽灵数据”。

由于 Discord 的帖子状态、反应数和内容会随时间发生变化（而机器人并不总能实时捕获所有事件），Auditor 通过后台轮询机制，确保本地索引与 Discord 服务端的数据达到**最终一致性**，并清理长期失效的“幽灵数据”。

---

## 🏗️ 核心组件

### 1. Auditor cog
审计器的核心入口，维护两个主要的异步任务循环：

- `audit_loop` (全量同步循环):
    - 加载：每轮开始时从数据库加载所有已索引的帖子 ID，并进行随机打乱。
    - 执行: 遍历队列，将同步任务提交至 `APIScheduler`。
    - 频率控制: 采用“慢速爬行”策略，每处理一个帖子后固定休眠 4 秒。
    - 优先级: 审计任务被赋予最低优先级，确保绝不影响用户的交互搜索请求。

- `cleanup_loop` (数据清理循环):
    - 逻辑: 负责物理删除本地数据库中已失效的记录。
    - 频率: 每 6 小时执行一次。
    - 阈值: 当一个帖子的 `not_found_count`（即同步时发现 404 的次数）达到 5 次时，该帖子被视为已被永久删除，并从数据库中物理移除。

### 2. `AuditorService`
数据访问层，封装了审计所需的 SQL 操作：
- 获取全量帖子 ID 列表。
- 根据阈值物理删除陈旧记录。

---

## ⚙️ 核心机制


### 同步与容错
当 `audit_loop` 调用 `sync_service.sync_thread(tid)` 拉取 Discord 数据时：
1. 如果帖子存在，更新本地的标题、内容摘要、反应数和标签。并将该帖子的 `not_found_count` 重置为 0。
2. 如果 Discord 返回 `NotFound`，`SyncService` 会使该帖子的 `not_found_count` 加 1。
3. 只要 `not_found_count > 0`，搜索模块就会自动过滤掉该帖子，直到它下次被成功审计。
